# ADR 1011 — The client/mobile Sentry scrubber strips URL query strings deny-by-default

**Status:** Accepted (2026-07-17) — **Skeleton-authored (channel-A drain of `de34dd2`, by content; ADR re-authored, not cherry-picked); HQ-ruled, Martin ratify queued.** Extends [ADR 0021](0021-telemetry-observability-package.md) (the telemetry scrub seam) and mirrors, on the client/mobile envelope, the leak class [ADR 1009](1009-sentry-request-pii-scrub.md) closed on the server (`apps/api/src/sentry/init.ts`).

## Context

`packages/telemetry/src/scrub.ts` is the **platform-neutral** PII scrubber that both Sentry bindings (`@repo/telemetry/web`, `@repo/telemetry/native`) wire into `beforeSend` / `beforeBreadcrumb` / `beforeSendTransaction` / `beforeSendSpan` via `buildSentryOptions`. It had two redaction mechanisms, both **deny-lists**:

1. **Value-shape patterns** — it redacts strings that _match a shape it enumerates_: Bearer/JWT/email/rodné-číslo.
2. **Sensitive key names** — it drops a value wholesale when its key matches the `pii()`-registry mirror (`email`, `authorization`, `name`, … plus the perimetra odběratel columns `ico`/`dic`/`phone`/`address_line`/`city`/`postal_code`, ADR 0071/0082).

An arbitrary querystring param is neither. A surname typed into a search box rides in `?search=Nováková`; the value `Nováková` has no matchable shape, and the param key `search` is not a `pii()` column, so **both** deny-lists pass it through. Concretely: a fetch/xhr breadcrumb carries `data.url = "https://app/api/clients?search=Nováková"`, and once `tracesSampleRate > 0` a span carries the same string in `data["url.full"]` / `description` and the bare querystring in `data["url.query"]`. This is the identical `?q=<pii>` leak class that ADR 1009 closed on the server (and that `stripQueryString` closes for pino) — but it reaches Sentry through the **client/mobile** envelope, which ADR 1009 does not touch. The closing recommendation of ADR 1009 named this exact follow-up: the client/mobile scrubber is pattern-based and misses a non-pattern querystring PII value in breadcrumbs/spans.

## Decision

Add a second, **deny-by-default** layer to the same scrubber: on URL-bearing fields, keep `scheme://host/path` and drop the query + fragment wholesale, rather than trusting a pattern to recognise the value. This is applied uniformly across every envelope the bindings use — the event/breadcrumb/transaction walk (`scrubValue`, reused by `scrubEvent` and `scrubBreadcrumb`, which `beforeSendTransaction` also calls) and the span scrubber (`scrubSpan`):

- **Always-URL fields** (`url`, `url.full`, `http.url`, `referer` / `referrer`, and the span attribute `http.request.header.referer`) → `dropUrlQuery` cuts at the first `?` or `#`, keeping origin + path; the surviving path still runs through pattern redaction, so a token baked into the **path** (`/reset/<jwt>`) is still caught by layer 1. `referer`/`referrer` are included because Sentry's default browser `httpContextIntegration` writes `document.referrer` — the full referring URL, query and all — into `event.request.headers.Referer` (and the span attribute) on **every** error and transaction.
- **Ambiguous path fields** (`to` / `from`, a navigation breadcrumb's paths) → query-stripped **only when the value is URL/path-shaped** (`http(s)://`, `//`, or a leading `/`). `to`/`from` are also generic key names for free text (confirmation copy, chat), so an unconditional strip would truncate a value like `"Cancel? yes"` at the `?`; the shape gate prevents that data corruption.
- **Bare-query fields** (`url.query`, `http.query`, `query_string`, `search`) have no path worth keeping → the whole value is dropped to `[Filtered]`.
- **Span `description`** (handled both by `scrubSpan` and — because a transaction event embeds `spans[]` and `contexts.trace`, walked by `scrubEvent` not `scrubSpan` — by a `description`-key case in the event walk) is query-stripped only when it is an HTTP request line (`GET https://…?q=…`); a `db.query` description (`… WHERE id = ?`) is left intact so its bind-placeholder `?` is not truncated.
- **Embedded absolute URLs in free text** (`event.message`, an `exception.value`, a breadcrumb string) → `redactString` itself now cuts the `?query`/`#fragment` off any `http(s)://…` substring, so the `?search=<surname>` class is closed even where no URL-named key reaches it. Only absolute URLs are touched (the `://` requirement) so a bare `1/2?x` in prose is never mistaken for one.
- **Nested objects under a span `data` attribute** are recursed into with the same key rules as the event walk (a data value is not passed through raw), closing the asymmetry where a nested `{email,…}` under a custom attribute would have shipped unredacted.

**The path is deliberately kept, not deleted.** ADR 1009's server scrubber `delete`s `event.request.url` down to nothing because a server error report does not need the route. The client/mobile scrubber keeps the path because a client trace is only debuggable with the route (`GET /api/clients` vs `POST /api/orders`), and Sentry's client tracing groups spans by it. Deny-by-default is applied to the query — the part that carries free-text user input — not to the whole URL.

**`scrubSpan` moved from `sentry-options.ts` into `scrub.ts`.** It previously lived inline in the options builder. Homing it beside the event walk single-homes every scrub rule (key drop, URL query strip, pattern redaction), so a change cannot be applied to one hook and missed on another, and it keeps `sentry-options.ts` in lockstep with the skeleton's (both now import `scrubSpan` from `./scrub`).

## Consequences

- The reported leak is closed on the client/mobile envelope across breadcrumb, span, and transaction — the surface ADR 1009 left open.
- **Behaviour change, deliberately strengthened:** a URL field is no longer rewritten to contain a `[Filtered]` marker — its query simply disappears and the path remains. Two `sentry-options.test.ts` assertions that expected a `[Filtered]` marker inside a scrubbed URL were updated to assert the exact path-kept/query-dropped result; the change makes the scrub stronger (an unmatched param is now removed, where before only matched shapes were), not weaker.
- **Telemetry only.** `scrub.ts` is a pure, SDK-free function on the error/trace-reporting path; it changes no stored data and no request/response the app serves — no RLS, migration, or contract surface.
- **The `pii()`-registry mirror is a perimetra superset and is unchanged by this ADR.** Perimetra's `SENSITIVE_KEYS` carries the odběratel columns (`ico`/`dic`/`phone`/`address_line`/`city`/`postal_code`) on top of the skeleton set; this change is purely additive to the URL layer and touches neither the mirror nor the value-shape patterns. `scrub.pii-contract.test.ts` continues to guard the mirror against registry drift.
- Coverage: `scrub.test.ts` and `sentry-options.test.ts` gain cases for the fetch-breadcrumb leak (the reported class), the Referer header + span referer attribute, `spans[].description` / `contexts.trace.description` in a transaction event, an embedded URL in a free-text message, `to`/`from` URL-shape gating (stripped when a path, untouched when free text), a url-keyed array, a nested object under a span `data` key, bare `query_string`/`url.query`/`search` drop, the path-token composition (query dropped **and** a JWT in the surviving path redacted), and the SQL-`?` (incl. `DELETE FROM`) non-truncation. Each is non-vacuous — removing a branch reddens a case.
- **The `referer`/`referrer` surface is CLOSED (an earlier upstream draft wrongly scoped it out).** Adversarial review upstream showed the referer field carries the whole **query**, exactly the `?search=` class this ADR targets, and that it was reaching Sentry unredacted — so it is now an always-URL field (path kept, query dropped), consistent with the server's ADR 1009 which deletes it wholesale.
- **Out of scope, recorded as owed:** `@repo/telemetry`'s scrubber is one of two client telemetry sinks — PostHog's `posthog-js` autocaptures a `$pageview` whose `$current_url` includes the query and does **not** pass through this scrubber. Closing that is a separate PostHog-config change (`@repo/telemetry/posthog-analytics`), tracked as a perimetra-owed follow-up, not this ADR.
